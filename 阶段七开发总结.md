# 阶段7开发总结（短信登录）

> 本阶段目标是把短信登录从“验证码能力可复用”推进到“真实服务商可配置、可发送、可登录、可测试”的完整闭环。

## 1. 阶段目标与完成情况

### 阶段目标
- 目标1：完成短信验证码登录主链路（发送验证码 -> 校验验证码 -> 登录/建号）。
- 目标2：完成短信服务商抽象与首版服务商接入。
- 目标3：预留第二家服务商扩展能力并补齐配置说明与测试。

### 完成结果（对照目标逐条写）
- 已完成：`SmsProvider` 已支持手机号验证码发送、验证码登录、新用户自动注册与老用户直登。
- 已完成：短信发送器已抽象为统一 `MessageSender` 体系，支持按 `senderName` 选择服务商。
- 已完成：首版接入阿里云短信发送器（`aliyun_sms`）。
- 已完成：预留并落地第二家服务商腾讯云短信发送器（`tencent_sms`）。
- 已完成：补齐 `auth.config.example.ts` 与 `.env.example` 的签名、模板、密钥等配置说明示例。
- 已完成：补齐腾讯云短信发送器单元测试，阶段内关键命令 `typecheck/build` 均通过。

### DoD 自检结论
- [x] DoD 条目1：用户完成短信服务商申请后，仅通过配置即可启用短信登录。
- [x] DoD 条目2：短信验证码发送、校验、登录、自动注册链路可用。
- [x] DoD 条目3：至少 2 家服务商具备可扩展结构与自动化测试覆盖。

## 2. 使用了什么技术，实现了什么功能

### 后端实现
- 技术/库：Node.js、TypeScript、Express、PostgreSQL、`@alicloud/dysmsapi20170525`、`@alicloud/openapi-client`、`tencentcloud-sdk-nodejs-sms`（可选 peer）。
- 核心实现点：
  - `SmsProvider` 复用 `VerificationService`，统一处理验证码生成、频控、过期与校验。
  - `MessageSenderRegistry` 统一管理短信/邮件发送器，按配置动态选择服务商。
  - `AliyunSmsMessageSender` 与 `TencentSmsMessageSender` 均采用懒加载 SDK + 错误码统一封装。
  - 腾讯云发送流程新增手机号格式规范化（E.164 场景下优先 `+86` 前缀）。
- 对应功能结果：短信登录具备“服务商可切换、链路可回归、配置可落地”的交付形态。

### 前端实现
- 技术/库：本阶段未新增前端页面或组件。
- 核心实现点：无。
- 对应功能结果：前端可继续通过既有认证路由接入短信登录。

### 数据/任务/接口实现（按阶段实际选填）
- 数据模型/迁移：
  - 复用 `verification_tokens` 表记录短信验证码。
  - 复用 `users` / `identities` 表完成短信身份登录与首登建号。
- 任务流转：
  - 请求验证码：`phone` 校验 -> 创建验证码 token -> 发送短信。
  - 验证登录：校验验证码 -> 查找短信身份 -> 已有用户登录/新用户建号。
- API 清单：
  - 复用认证体系下短信 Provider 对应登录接口（由现有 Adapter 路由接入）。

## 3. 新使用技术栈简要介绍

> 本阶段新增重点在“可选依赖 + 服务商扩展”，避免把单一服务商能力写死。

- `tencentcloud-sdk-nodejs-sms`：用于腾讯云短信发送，作为可选 peer 依赖引入，避免非腾讯用户被强制安装。
- 可选 SDK 类型声明（`optional-sdk-modules.d.ts`）：用于在 TypeScript 下兼容动态导入的可选依赖，保证 `typecheck` 稳定。
- 发送器懒加载模式：仅在真实发送时加载短信 SDK，降低初始化开销并减少无关环境报错。

## 4. 遇到的问题与解决方案

### 问题1：可选 SDK 导致编译期找不到模块类型
- 现象：`typecheck/build` 报错 `Cannot find module 'tencentcloud-sdk-nodejs-sms'`。
- 原因：SDK 采用动态导入且为可选依赖，默认无类型声明。
- 解决：新增 `src/types/optional-sdk-modules.d.ts` 的腾讯云模块声明，并将 SDK 设为 `peerDependencies + optional`。
- 防复发措施：后续新增可选服务商 SDK 时，必须同步补类型声明与可选 peer 配置。

### 问题2：短信服务商配置入口不统一，接入成本高
- 现象：接入方难以直观看到短信签名、模板、密钥字段。
- 原因：示例配置中缺少腾讯云短信完整样例。
- 解决：补齐 `auth.config.example.ts` 与 `.env.example` 的腾讯云配置字段。
- 防复发措施：每新增 Provider，必须同时更新配置示例和环境变量模板。

### 问题3：缺少腾讯短信发送器自动化测试
- 现象：改动后只能靠人工推断正确性。
- 原因：未覆盖腾讯短信成功/失败/错误通道场景。
- 解决：新增 `tests/unit/tencent-sms-message-sender.test.ts`，覆盖 3 条关键路径。
- 防复发措施：服务商发送器必须至少具备“成功 + 失败 + 参数校验”三类单测。
